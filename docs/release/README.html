README

JSwat Debugger

JSwat is a graphical Java debugger front-end, written for the Java Platform, utilizing the Java Platform Debugger Architecture (JPDA), and the NetBeans Platform. The source code is licensed under the Common Development and Distribution License.

Requirements

Java SE with JPDA

JSwat requires the Java Platform, Standard Edition, version 6.0 or higher. It is possible, although not necessarily recommended, to use compatible software from other vendors. In any case, the selected Java SE must contain the JPDA software. The required classes, defined in the com.sun.jdi package, are typically found in a file called tools.jar.

Note that the Java Runtime Environment (JRE) does not contain the JPDA and therefore will not run JSwat properly. You must use the Java Development Kit (JDK) to run JSwat.

Compile for debugging

When debugging Java code, it is necessary to compile the classes using the -g option to javac. Additionally, do not compile the classes with the optimization flags, as that may hinder debugging. If you use Apache Ant to compile your application, set the debug value of the javac task to on and either leave the debuglevel value undefined, or use the value vars,lines,source so as to generate all of the necessary debugging information in the compiled classes.

Starting JSwat

Starting the debugger amounts to invoking a small launcher. There are two methods for this and the choice depends on your system.

Microsoft Windows

If you are using Microsoft Windows, you can double-click on the jswat.exe launcher in the bin directory, or invoke it from the command line.

Unix-like systems

On Unix-like systems such as BSD, Linux, Mac OS X, or Solaris, simply invoke the shell script jswat in the bin directory.

Startup options

The JSwat launcher is basically a renamed version of the NetBeans Platform launcher, thus it supports the same command-line options. For a list of these options and a brief explanation, invoke the launcher with the --help switch.

Documentation

The JSwat documentation consists of a User's Guide, the built-in help found in the Help menu in the main program window, and in several "how to" documents on the web site.

Stopping JSwat

The correct way to exit JSwat is with either the window close button or the Exit item in the File menu. Using Ctrl-c to kill JSwat will likely cause problems and should be avoided. It must only be used in cases where the JVM hosting JSwat has frozen. If the JSwat interface appears frozen, chances are that terminating the debuggee, via the appropriate system operation, will allow JSwat to return to normal.

Removing JSwat

Removing JSwat from your system is generally pretty simple. Find the files, wherever it is that you originally put them, and delete them. If you used one of the packaged versions you should remove the package using the appropriate system command. In addition to the program files, there are user settings stored in a directory named .jswat in your home directory.

Upgrading from earlier versions

The settings from the 2.x versions of JSwat may be safely removed — in other words, they are ignored by version 3.0 and later. These resided in one of two places, depending on which operating system you are using.

  • Under .java/.userPrefs/com/bluemarsh/jswat/ in your Unix home directory.
  • Under HKEY_CURRENT_USER/Software/JavaSoft/Prefs/com/bluemarsh/jswat in the Windows registry.

The 1.x versions of JSwat stored their settings in a directory called .jswat in your home directory. However, the 3.x versions of JSwat also store their settings in this directory, so you should not simply delete this directory. Instead, you can remove the 1.x files only, leaving the rest intact. They are named breakpoints, session, and settings and can be safely removed.

Upgrading the JDK

If at some point you replace your current JDK installation with a different version, you will need to update the JSwat configuration file. This is done by opening the etc/jswat.conf file in a text editor and changing the netbeans_jdkhome value to point to the new JDK installation directory. An alternative to editing the configuration file is to pass the --jdkhome option when starting JSwat. This option takes a single argument, which is the path to the JDK installation.

Getting source code

The source code for each release of JSwat is distributed via the same web site as the binary download. To get the very latest source code you must use a Git client to access the source repository. See the project web site for more information on utilizing the source code repository.

Troubleshooting

Debugging Features Disappeared

There are multiple causes for this behavior.

  1. The JSwat modules require Java 6.0 or higher, and when the debugger is launched using an earlier JDK, the JSwat modules will automatically be disabled. To rectify the situation, you will need to exit and start again using JDK 6.0 or higher (see section above regarding upgrading the JDK), and then re-enable the JSwat modules. This is done using the Plugins manager which is found in the Tools menu. Check the boxes next to each of the JSwat modules to enable them. Everything should be back to normal at that point, allowing you to debug using JSwat.
  2. If the debugger used to work and now appears to have lost all of the debugging features, it may be that the software packages on your system have been updated and created an incompatibility. One possible solution is to (re)move the ~/.jswat directory and start JSwat again. You will lose your serialized settings, but chances are that will resolve the problem.

Slow Performance on Linux

Due to a change in Linux kernel 2.6.15, the Sun JPDA implementation is adversely affected in terms of network performance. This causes all Java debuggers based on JPDA to perform very, very slowly. There are a few possible solutions to this problem.

  • Downgrade to kernel 2.6.14 or lower
  • Upgrade to JDK 6.0 build 79 or higher
  • Invoke sudo sysctl -w net.ipv4.tcp_abc=0 to effectively reverse the change made in the Linux kernel. Note that this may incur a security risk; speak with your system administrator if you are at all concerned.
